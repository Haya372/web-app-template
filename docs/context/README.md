# context/

このディレクトリには、このリポジトリが想定している **前提・背景・ドメイン知識・用語** をまとめます。

設計判断の理由は `../decisions/` に、開発ルールは `../guidelines/` に記載します。

## ディレクトリ構成

```
context/
├── README.md          # このファイル（運用ルール）
├── glossary.md        # プロジェクト全体のユビキタス言語
└── domains/
    ├── _template.md   # ドメインドキュメントの雛形
    └── <entity>.md    # エンティティ/集約単位のドメイン知識
```

## domains/ に書くこと

- **課題 / 目的**: そのドメインが解決しようとしている問題
- **業務ルール / 制約**: 変わりにくい概念レベルのルール（例: ステータス遷移の制限、不変条件）
  - ビジネスの不変条件や状態遷移のみ記載する
  - 実装パターン（「別エンティティとして実装する」など）は含めない
- **用語**: そのドメイン固有の語彙と定義

## domains/ に書かないこと

- UseCase の手順 → コードを読む（`go-backend/internal/usecase/`）
- 値オブジェクトの実装詳細 → コードを読む（`go-backend/internal/domain/`）
- API 仕様 → `openapi/` を参照
- 設計判断の理由 → `../decisions/`（ADR）

## glossary.md と domains/ の使い分け

| 場所 | 対象 |
|------|------|
| `glossary.md` | 2 つ以上のドメインで出現する用語 |
| `domains/<entity>.md` の「用語」 | 1 つのドメインでのみ出現する用語、またはドメイン固有の解釈を持つ用語 |

**判断基準**: 用語が 2 つ以上のドメインで出現する場合は `glossary.md` に掲載する。

例:
- 「User」「認証」「注文」など複数ドメインに出現する用語 → `glossary.md`
- 「認可スコープ」など Auth ドメインでのみ意味を持つ用語 → `domains/auth.md` の「用語」
- `glossary.md` に掲載済みの用語でも、ドメイン固有の解釈がある場合は `domains/<entity>.md` でも記載してよい

## 更新のタイミング

**更新する（ドメイン知識の変更）:**
- ドメインの概念（エンティティ / 集約境界）が変わった
- 業務ルール（不変条件・状態遷移）が変わった
- 用語が追加・変更された

**更新しない（コードの変更のみ）:**
- 純粋な内部リファクタリング（ドメインルール不変）
- UseCase 層の実装追加のみ
- API エンドポイントの追加のみ

## 新しいドメインを追加するとき

1. `domains/_template.md` をコピーして `domains/<entity>.md` を作成
2. 新規用語を `glossary.md` に追加
3. PR 本文の「ドメイン知識更新」セクションで参照
