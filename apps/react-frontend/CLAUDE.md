# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Run from `apps/react-frontend/`:

```bash
# Development server (http://localhost:3000)
pnpm dev

# Type checking
pnpm typecheck

# Generate route tree (run after adding/removing route files)
pnpm generate:route

# Format and lint
pnpm fmt          # biome lint --write + knip --fix
pnpm lint         # biome lint + knip (read-only check)
pnpm check        # biome check (format + lint combined)

# Tests
pnpm test         # vitest run (single pass, no DB needed)

# Build
pnpm build        # prebuild (generate + typecheck) then vite build
pnpm preview      # preview the production build locally
```

## Design Principles

- Keep components small and focused on a single responsibility
- Prefer composition over inheritance; share logic via custom hooks
- Follow TDD where applicable: write a failing test, implement the minimum to pass, then refactor
- Keep PRs small with clear motivation, scope, and test evidence

## Architecture

The app uses TanStack Start (SSR-capable React meta-framework) built on TanStack Router:

```
src/
  routes/          # File-based routing — one file per route
    __root.tsx     # Root layout: HTML shell, Header, Footer
    index.tsx      # Home page (path: /)
    about.tsx      # About page (path: /about)
  features/        # Feature-based modules (co-locate components/hooks/api/types/pages per feature)
    <feature>/
      components/  # Components used only by this feature
      hooks/       # Hooks used only by this feature
      api/         # API client functions for this feature
      types/       # Domain and response types for this feature
      pages/       # Full-page components rendered by route files
  components/      # Shared UI components used across multiple features (PascalCase)
  hooks/           # Shared hooks used across multiple features
  utils/           # Pure utility functions (no side effects)
  styles.css       # Global styles and CSS custom properties (Tailwind v4)
  router.tsx       # Router instantiation and type registration
  routeTree.gen.ts # Auto-generated route tree — do not edit manually
```

**Key constraints:**

- Route files live under `src/routes/`. The file path maps directly to the URL path.
- `routeTree.gen.ts` is auto-generated by `pnpm generate:route` (TanStack Router CLI). Never edit it manually.
- Use the `@/*` alias for all cross-feature and shared-module imports (e.g., `import Header from '@/components/Header'`). Relative paths are fine within the same feature directory.
- Import files directly by path — both within a feature and across features. `index.ts` barrel re-exports are forbidden.
- Build app screens by composing `packages/ui` components. Never pass `className` to a `packages/ui` component (use `variant`/`size` props instead). For layout adjustments, wrap the component and apply Tailwind classes to the wrapper.
- Tailwind CSS v4 is used for styling; custom design tokens are defined as CSS variables in `src/styles.css`.

## Routing Rules (TanStack Router)

- **File naming:** `src/routes/<path>.tsx` → URL `/<path>`. Nested: `src/routes/posts/$id.tsx` → `/posts/:id`.
- **Layout routes:** use `__root.tsx` for the global shell. Nested layouts use `_layout.tsx` naming.
- **Route export:** every route file must export `const Route = createFileRoute('<path>')({ ... })`.
- **Regenerate after changes:** always run `pnpm generate:route` after adding, removing, or renaming route files so `routeTree.gen.ts` stays in sync.
- **Type-safe links:** use `<Link to="/about" />` (from `@tanstack/react-router`) rather than `<a href>` for internal navigation.
- **Loaders:** use `Route.loader` for data fetching before render; do not fetch inside `useEffect` for critical data.

## Coding Style

- Use Biome for formatting (tabs, double quotes for JSX strings) and linting; always pass `pnpm lint` before pushing.
- Use `@/*` import alias for cross-feature and shared-module imports; relative paths are fine within the same feature directory.
- Component filenames and function names use **PascalCase** (`Header.tsx`, `function Header()`).
- Non-component TypeScript files use **camelCase** (`router.tsx`, `useTheme.ts`).
- Hooks must start with `use`; place in `src/features/<feature>/hooks/` if feature-specific, or `src/hooks/` if shared.
- `any` is forbidden; use `unknown` and narrow with type guards or zod. `as` casts are strongly discouraged — `as any` is unconditionally forbidden.
- Avoid inline styles; express all visual variations via Tailwind utility classes or CSS variables.

## Testing

- Use **Vitest** with **jsdom** environment for unit and component tests.
- Place test files adjacent to the source file: `Header.test.tsx` next to `Header.tsx`.
- Use table-driven tests for utility functions with boundary and error cases.
- There is no integration test setup; focus unit tests on pure logic and component rendering.
- Aim for meaningful coverage of business logic; UI snapshot tests are discouraged.

## Security

- Never hardcode API endpoints, tokens, or secrets — read them from environment variables (`import.meta.env.*`).
- Sanitize user-supplied content before rendering to prevent XSS attacks.
- Use `rel="noopener noreferrer"` on all `target="_blank"` external links.
